## 1. 项目全景图

### 1.1 项目定位

这是一个**桌面端视频标注工具**，核心用途是：

- 加载视频文件，逐帧浏览
- 手动或 AI 辅助为视频片段生成描述标签
- 将标注好的片段导出为独立视频文件 + 对应文本标签

适用于视频数据集构建、内容审核、短视频打标签等场景。

## 2. 核心架构模式：Mixin 模块化

这是整个项目最核心的设计。理解了这个，你就理解了项目 60% 的结构。

### 2.1 什么是 Mixin？

Mixin 是一种代码复用模式：把一组函数定义在独立模块中，然后在类体内用 `from module import (func1, func2, ...)` 导入。这些函数虽然定义在模块顶层，但因为它们的第一个参数是 `self`，导入后**表现得像类的方法**。

### 2.2 在代码中的体现

看 `video_tagger.py` 第 297-328 行：

```python
class VideoTagger:
    # ... __init__ 和 setup_ui 在上面 ...

    # 导入模块 —— 关键代码
    from ui_events import (
        clear_frame_marks, delete_tag, edit_tag, ...
    )
    from video_processing import (
        load_video, load_video_manager, preprocess_frames, ...
    )
    from tag_management import (
        add_tag, export_tags, save_tag_records, ...
    )
    from ai_features import (
        generate_ai_caption, auto_segment_and_recognize, ...
    )
    from presets import (
        create_preset_item, use_preset_tag, ...
    )
    from utils import (
        is_child_of, draw_tag_markers, highlight_tag_for_current_frame
    )
```

这些导入的函数**直接变成了 `VideoTagger` 实例的方法**。调用时和写在本类里完全一样：

```python
self.load_video()       # 实际调用的是 video_processing.load_video(self)
self.add_tag()          # 实际调用的是 tag_management.add_tag(self)
self.draw_tag_markers() # 实际调用的是 utils.draw_tag_markers(self)
```

### 2.3 为什么这样设计？

| 优点 | 缺点 |
|------|------|
| 功能拆分清晰，每个模块负责一个领域 | 所有模块共享同一个 `self`，耦合度高 |
| 主文件 `video_tagger.py` 保持简洁 | IDE 代码跳转可能不如定义在类内直观 |
| 新增功能只需加一个模块并 import | 调试时 stack trace 跨多个文件 |
| 各模块可独立测试（函数不依赖类定义） | 修改 `self` 上的状态变量名会影响所有模块 |

### 2.4 共享状态（self 上的变量）

所有 Mixin 模块都通过 `self` 访问和修改同一组状态。这些变量是模块之间沟通的**唯一桥梁**：

| 变量 | 类型 | 含义 | 主要使用者 |
|------|------|------|-----------|
| `self.root` | `tk.Tk` | 主窗口 | 所有模块 |
| `self.video_path` | `str` | 当前加载的视频文件路径 | video_processing |
| `self.cap` | `cv2.VideoCapture` | 视频捕获对象，None 表示未打开 | video_processing |
| `self.total_frames` | `int` | 视频总帧数 | 所有模块 |
| `self.fps` | `float` | 视频帧率 | 所有模块 |
| `self.current_frame` | `int` | 当前帧索引 | ui_events, video_processing |
| `self.playing` | `bool` | 是否正在播放 | ui_events |
| `self.start_frame` | `int` | 标记的开始帧 | tag_management, ui_events |
| `self.end_frame` | `int` | 标记的结束帧 | tag_management, ui_events |
| `self.tags` | `list[dict]` | 标签列表 | tag_management, presets |
| `self.excluded_segments` | `list[dict]` | 排除片段 | ai_features, tag_management |
| `self.processed_frames` | `list[np.ndarray]` | 预处理后的帧 | video_processing, ai_features |
| `self.frames_loaded` | `bool` | 帧是否加载完成 | video_processing |
| `self.caption_presets` | `list[dict]` | AI 预设（带缩略图） | presets, ai_features |
| `self.manual_presets` | `list[str]` | 手动预设 | presets |
| `self.config` | `ConfigParser` | 配置对象 | ai_features, video_processing |
| `self.ai_prompt_entry` | `tk.Text` | AI 提示词输入框 | ai_features, ui_events |
| `self.export_fps` | `tk.StringVar` | 导出帧率 | tag_management |
| `self.export_dir` | `str` | 导出路径 | tag_management |

**阅读任何模块的函数时，先搞清楚它读取/修改了 `self` 上的哪些变量，逻辑就清晰了一半。**

#### 3.2.2 `preprocess_frames` — 核心帧预处理

这是整个视频加载过程中**最耗时**的一步。它做了三件事：

**步骤 1：创建进度对话框**

```python
self.loading_window = tk.Toplevel(self.root)
self.loading_window.grab_set()  # 模态化，阻止用户操作主窗口
```

**步骤 2：逐帧读取、下采样、缩放**

```python
target_fps = self.config.getint('PROCESSING', 'target_frame_rate', fallback=24)

# 计算降采样间隔
if self.fps > target_fps:
    frame_interval = self.fps / target_fps  # 例：原 30fps → 30/16 = 1.875
    effective_fps = target_fps
else:
    frame_interval = 1
    effective_fps = self.fps

i = 0
while True:
    ret, frame = self.cap.read()
    if not ret: break

    if i % frame_interval < 1:  # 关键采样逻辑
        frame = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)  # BGR → RGB
        frame = self.resize_to_720p(frame)              # 等比缩放到 720p
        self.processed_frames.append(frame)             # 存入内存
```

**采样逻辑详解**：

假设原视频 30fps，目标 16fps，`frame_interval = 30/16 ≈ 1.875`：

| i | i % 1.875 | 结果 |
|---|-----------|------|
| 0 | 0 | 保留 |
| 1 | 1.0 | 保留 |
| 2 | 0.125 | 保留 |
| 3 | 1.125 | 跳过 |
| 4 | 0.25 | 保留 |
| 5 | 1.25 | 跳过 |

这本质上是**均匀下采样**，效果接近每隔 1.875 帧取一帧。

**步骤 3：更新状态**

```python
self.frames_loaded = True
self.total_frames = processed_frame_count   # 更新为实际处理后的帧数
self.fps = effective_fps                    # 更新为实际有效帧率
self.export_fps.set(f"{self.fps:.2f}")      # 默认导出帧率
self.show_frame()
self.draw_tag_markers()
```

#### 3.2.3 `resize_to_720p` — 等比缩放

```python
def resize_to_720p(self, frame):
    target_height = self.config.getint('PROCESSING', 'target_frame_height', fallback=720)
    h, w = frame.shape[:2]
    if h <= target_height:
        return frame  # 已经是小分辨率
    new_height = target_height
    new_width = int(w * (new_height / h))  # 保持宽高比
    return cv2.resize(frame, (new_width, new_height))
```

#### 3.2.4 `show_frame` — 帧渲染到画布

```python
def show_frame(self):
    frame = self.processed_frames[self.current_frame]
    image = Image.fromarray(frame)  # numpy array → PIL Image

    canvas_width = self.video_canvas.winfo_width()
    canvas_height = self.video_canvas.winfo_height()
    if canvas_width <= 1: canvas_width = 800   # 处理初始化时尺寸为 1 的情况
    if canvas_height <= 1: canvas_height = 600

    scale = min(canvas_width / img_width, canvas_height / img_height)
    if scale > 1: scale = 1  # 不放大

    resized_image = image.resize((new_width, new_height), Image.LANCZOS)
    photo = ImageTk.PhotoImage(resized_image)

    self.video_canvas.create_image(x, y, anchor=tk.NW, image=photo)
    self.video_canvas.image = photo  # ⚠️ 保持引用，防止 GC 回收导致显示空白

    self.progress.set(self.current_frame)
    self.frame_label.config(text=f"帧：{self.current_frame}/{self.total_frames-1}")
```

> `self.video_canvas.image = photo` 这一行非常重要。Tkinter 的 PhotoImage 如果没有被变量持有引用，会被 Python 垃圾回收，画布变空白。

#### 3.2.5 `play_video` — 播放循环

```python
def play_video(self):
    if not self.playing or not self.frames_loaded:
        return
    self.current_frame += 1
    if self.current_frame >= self.total_frames:
        self.current_frame = 0  # 循环播放
    self.show_frame()
    self.draw_tag_markers()
    self.root.after(int(1000/self.fps), self.play_video)  # 安排下一次
```

**工作原理**：`root.after(ms, callback)` 是 Tkinter 的定时器。fps=16 时每隔 62ms 调用一次，形成动画循环。这是**事件循环驱动**的方式，不是 `while True` 死循环——UI 依然可以响应其他事件。

#### 3.2.6 `save_and_replace_video` — 裁剪替换

用 start_frame 到 end_frame 区间生成新视频，删除原文件后重命名。使用**临时文件**策略（先写入 `_temp_xxx.mp4`，成功后再删除原文件），避免中途失败导致原文件丢失。

#### 3.3.4 `export_tags` — 导出片段

这是工具最关键的输出功能：

```python
for i, tag in enumerate(self.tags):
    # 1. 获取第一帧确定输出尺寸
    first_frame = self.processed_frames[start_frame]
    height, width = first_frame.shape[:2]

    # 2. 生成安全文件名（截取前 10 个有效字符）
    safe_tag_text = sanitize(tag["tag"])[:10]
    filename = f"video_{i+1:03d}_{safe_tag_text}"

    # 3. 处理文件名冲突
    while os.path.exists(video_path):
        filename = f"video_{i+1:03d}_{safe_tag_text}_{counter}"
        counter += 1

    # 4. 创建 VideoWriter，逐帧写入（RGB → BGR）
    out = cv2.VideoWriter(video_path, fourcc, export_fps, (width, height))
    for frame_num in range(start_frame, end_frame + 1):
        frame_bgr = cv2.cvtColor(self.processed_frames[frame_num], cv2.COLOR_RGB2BGR)
        out.write(frame_bgr)
    out.release()

    # 5. 写入配套 .txt 文件
    with open(txt_path, 'w', encoding='utf-8') as f:
        f.write(tag_text)
```

**帧采样 `extract_frames`**：

```python
def extract_frames(self, processed_frames, start_frame, end_frame):
    max_sample_frames = self.config.getint('PROCESSING', 'max_sample_frames', fallback=64)
    total_frames = end_frame - start_frame + 1
    sample_count = min(max_sample_frames, total_frames)

    if total_frames <= sample_count:
        indices = list(range(start_frame, end_frame + 1))  # 全取
    else:
        indices = np.linspace(start_frame, end_frame, sample_count, dtype=int)

    frames = [Image.fromarray(processed_frames[i]) for i in indices]
    if len(frames) == 1:
        frames.append(frames[0])  # 至少 2 帧
    return frames
```

> `np.linspace(0, 999, 64)` 生成 `[0, 15, 31, 47, ..., 984, 999]`，均匀间隔采样。

**图片编码 `convert_image_to_base64`**：

```python
def convert_image_to_base64(self, image):
    image.thumbnail((720, 720), Image.Resampling.LANCZOS)
    buffered = io.BytesIO()
    image.save(buffered, format="PNG")
    encoded = base64.b64encode(buffered.getvalue()).decode("utf-8")
    return f"data:image/png;base64,{encoded}"
```

**核心方法 `generate_caption`**：

```python
def generate_caption(self, frames, prompt_text=None, max_attempts=10, min_length=50):
    # 1. 所有帧转 base64
    image_data_urls = [self.convert_image_to_base64(frame) for frame in frames]

    # 2. 构造多模态消息
    content_list = [{"type": "text", "text": prompt_text or default_prompt}]
    for data_url in image_data_urls:
        content_list.append({"type": "image_url", "image_url": {"url": data_url}})
    messages = [{"role": "user", "content": content_list}]

    # 3. 调用 API（带重试逻辑）
    for attempt in range(max_attempts):
        response = self.client.chat.completions.create(
            model=model_name, messages=messages,
            max_tokens=max_new_tokens, temperature=temperature, top_p=top_p
        )
        caption = response.choices[0].message.content.strip()

        # 4. 清理推理模型的思考过程（<thought>...<thought> 标签）
        thought_pattern = r"<thought>.*?</thought>"
        cleaned_caption = re.sub(thought_pattern, "", caption, flags=re.DOTALL | re.IGNORECASE).strip()
        if cleaned_caption:
            caption = cleaned_caption

        # 5. 检查过滤词
        contains_filter_word = any(word in caption.lower() for word in filter_words)
        if contains_filter_word and attempt < max_attempts - 1:
            continue  # 包含过滤词，重新生成

        # 6. 检查长度
        if len(caption) < min_length:
            if attempt < max_attempts - 1:
                continue
            return "视频描述内容过短，无法提供有效描述"

        if not contains_filter_word:
            return caption
```

**重试触发条件**：
1. 返回内容包含过滤词（如 "blurry", "low resolution", "无法提供" 等）
2. 返回内容少于 `min_length` 字（默认 50）
3. API 调用异常（网络超时等）

最多重试 10 次（自动分段时设为 20 次，min_length 设为 100）。

## 6. 关键设计决策与取舍

### D1: 为什么所有帧都缓存在内存中？

**原因**：实现简单。`processed_frames` 是一个 Python list，`processed_frames[i]` 就是第 i 帧，随机访问 O(1)。不需要 Seek、不需要维护解码状态。

**代价**：内存占用大。一段 10 分钟 1080p 视频可能占用 1-2 GB。

**可选改进**：按需解码 + LRU 缓存（只保留最近访问的 N 帧）。

### D2: 为什么用 Mixin 而不是继承？

Mixin 比继承更灵活——不需要多层继承链，模块之间是平级的。每个模块只关注一个功能域，用 `from module import *` 注入即可。

### D3: 为什么 AI 调用用 OpenAI SDK 而不是 requests？

OpenAI SDK 内置了超时、重试、流式处理等机制，而且 API 格式标准化。换用任何兼容 OpenAI 接口的本地服务（LM Studio、Ollama、vLLM）只需改 `base_url`。

### D4: 为什么 UI 更新必须用 root.after？

Tkinter **不是线程安全的**。所有 UI 操作（创建窗口、修改文本、更新列表）必须在主线程中执行。后台线程调用了 AI API 得到结果后，必须通过 `root.after(0, callback)` 把 UI 更新调度到主线程。

### D5: 为什么有重复的 AI 生成代码？

`tag_management.py` 中的 `_generate_single_tag_caption` 和 `ai_features.py` 中的 `LLMClient.generate_caption` 逻辑重复。这是 AI 功能迭代过程中产生的——最初直接在 tag_management 中实现，后来抽象出 LLMClient 但没有清理旧代码。建议后续统一。

### D6: 为什么照片编码用 base64 而不是上传文件？

因为多模态 LLM API 通常期望图片以 base64 data URL 格式内嵌在消息中（OpenAI Chat Completions API 的 `image_url` 字段）。这避免了单独的文件上传步骤。

## 7. 常见扩展方向

### 7.1 内存优化

当前所有帧缓存在 `processed_frames` 列表中。可以改为：

- **快捷键自定义**：在设置中修改快捷键绑定
- **深色模式**
